# Best Practices Research

現在のタスクに対して **複数の検索角度** からベストプラクティスを収集し、「今やっていることと外部知識の接合」を行う。

---

## トリガー

### 自動トリガー
- 「ベストプラクティスは？」「他はどうやってる？」「参考事例を調べて」
- 「どういう設計がいいか迷ってる」「知識が足りないから調べて」
- 「一般的にはどうする？」「業界標準は？」

### 手動トリガー
- `/best-practices <テーマ>`
- `/bp <テーマ>`

---

## Flow

### Step 1: タスクコンテキストの抽出

args または直前の会話から以下を特定する:

```
- 主題: 何をやっているか（例: "Supabase RLSの設計"、"TypeScriptのエラーハンドリング"）
- スタック: 使っている技術・ツール・フレームワーク
- 制約: 既存コードベース・チーム規約・CLAUDE.mdのルール
- 不確実性: 何がわからないか・何を確認したいか
```

引数が与えられた場合はそれを主題とする。引数なしの場合は直前の会話から推定して **確認なしで進める**。

### Step 2: 検索クエリの生成

主題から以下の **6角度** でクエリを生成する。角度ごとに独立したクエリを1〜2本:

| 角度 | 狙い | クエリ例 |
|------|------|---------|
| **公式ドキュメント** | 正式な推奨方法 | `"<tech> best practices site:docs.xxx.com"` |
| **コミュニティ知見** | 実践者の経験 | `"<tech> best practices 2024 reddit OR stackoverflow"` |
| **失敗パターン** | アンチパターン・落とし穴 | `"<tech> common mistakes pitfalls avoid"` |
| **OSS実装例** | 生きたコード参照 | `"<tech> example implementation github"` |
| **比較・議論** | 選択肢の是非 | `"<tech> vs <alternative> tradeoffs"` |
| **日本語リソース** | 日本固有のコンテキスト（規制・慣習など） | `"<tech> ベストプラクティス OR 設計指針"` |

### Step 3: 並列Web検索

上記クエリを **並列** で実行する。

```
各クエリごとに:
1. WebSearch で上位5件を取得
2. 関連度が高い1〜2件の本文を WebFetch で読む
3. 重要ポイントを箇条書きで抜き出す
```

制限:
- 1クエリあたりのFetch上限: 2件
- 総Fetch上限: 8件
- タイムアウト: 各60秒

### Step 4: 合成と接合

収集した情報を **3層** で整理する:

#### Layer 1: 外部ベストプラクティス（What others do）
収集した知見を、重複排除・重要度順で整理。

#### Layer 2: 現在のコンテキストとのGAP分析
```
現状のアプローチ: [Step 1で把握した内容]
ベストプラクティスとの差分:
  ✅ 既にやっている（合致）
  ⚠️  部分的に合致、改善余地あり
  ❌ 未対応 / 逆行している
```

#### Layer 3: 推奨アクション
現在のタスクに即した **具体的な次のアクション** を提示。抽象論ではなく、このプロジェクト・このコードに対して何をすべきか。

### Step 5: 出力フォーマット

```markdown
## ベストプラクティス調査: {主題}

### 検索した角度
- 公式ドキュメント / コミュニティ知見 / 失敗パターン / OSS実装 / 比較議論 / 日本語リソース

---

### 外部ベストプラクティス（重要度順）

1. **{ポイント}**
   - 根拠: {出典URL or ソース名}
   - 要点: {2〜3行の説明}

2. ...

---

### 現在の作業との接合

| 項目 | 外部標準 | 現状 | 評価 |
|------|----------|------|------|
| {観点} | {標準的アプローチ} | {現在やっていること} | ✅/⚠️/❌ |

---

### 推奨アクション

優先度高:
- [ ] {具体的なアクション（コード・設定・設計変更）}

優先度中:
- [ ] ...

優先度低 / 将来検討:
- [ ] ...

---

### 参照ソース
- {URL}: {1行サマリー}
```

## 注意事項

- **情報の鮮度**: 検索結果の日付を確認し、2年以上古い場合は明示する
- **スタックバージョン**: バージョン固有の情報は必ずバージョンを明記する
- **意見の多様性**: コミュニティで意見が割れている場合は両論を提示し、判断はユーザーに委ねる
- **CLAUDE.mdとの整合**: 関数型・不変・型安全などの既存制約と矛盾するベストプラクティスは、制約側を優先して調整案を提示する
- **英語リソース主体でも日本語で報告**: 出典はURLで示し、説明は日本語

## このスキルがやらないこと

- 実装・コード変更の実行（調査と接合まで。実行はユーザーが承認後）
- Scrapboxへの自動書き込み（必要なら別途指示）
- 特定プロダクトへの依存（汎用技術調査ツールとして機能する）
