# 経費精算 (多元現実)

## 0. 大原則（最初に必ず読む）
- **情報・領収書・決済手段が揃っていなければ無理に進めない。** 不足ならファイル/レシート/決済手段を要求するか、`AskUserQuestion` で埋める。憶測で freee に書いたり Google フォームに出したりしない。
- **「領収書を見た → そのままフォームに出す」は禁止。必ず先に freee の法人口座明細を照合する。** 領収書の宛名が「多元現実」でも、その支払いを会社が既に負担している（法人カード直接 or 個人への前渡し振込）ことがある。照合を飛ばすと二重取りになる。← この skill が生まれた最大の理由。

## 1. 決済手段の3分類（これを最初に確定する）
| 分類 | 定義 | 正しい処理 | Gフォーム |
|---|---|---|---|
| **A 法人カード直接** | GMOあおぞらデビット(***8882)で直接決済済み | 立替ではない。freee に自動取込される。勘定科目（旅費交通費等）で登録するだけ | **不要** |
| **B 個人立替** | 個人のカード/現金で払い、会社からまだ受け取っていない | Gフォームで請求 → 経理が精算スプシで人別合計→送金 | **要** |
| **C 前渡し / 仮払** | 会社が先に個人口座へ振込み、その金で個人が支払った | 前渡し額=仮払金。実費を旅費交通費へ振替、差額(余り)を返金 or 次回相殺 | **不要（二重取り注意）** |

判別の勘所:
- 領収書の決済が「クレジットカード利用」でも、それが**法人カードなら A**、**個人カードなら B か C**。カード名義をユーザーに確認する。
- 出張の少し前に**会社→個人口座への振込**があれば C（前渡し）。これは領収書には現れない。freee 照合でしか分からない。

## 2. 標準手順（最小ステップ＝トークン節約）
1. **領収書から抽出**: `{日付, 金額, 宛名, 区間/内容, 決済手段}`。PDF/画像は `Read` で直接読む（ファイル名が中身と逆のことがある＝中身で判断）。
2. **freee 照合（必須・原則1回のАPI呼び出し）**: 出張期間の法人口座明細を取得し、A(直接決済)と C(前渡し振込)を検出する。
   ```
   GET /api/1/wallet_txns
     company_id=12669261, walletable_id=4772220,
     walletable_type=bank_account, start_date=..., end_date=...
   ```
   - 個人名（例「タカギシユンスケ」）への `振込` = **前渡し(C)**。
   - 領収書の店名と同じ `Visaデビット利用 ...` = **直接決済(A)**。`PENDING` と同額の `取消(income)` はペアで相殺＝実質ゼロ。
3. **分類を確定** → 表1の処理を実行。立替(B)だけが Gフォーム対象。
4. **判断が要る点だけ `AskUserQuestion`**（差額の扱い、Gフォーム要否の最終確認、勘定科目）。

## 3. freee 参照定数（多元現実・ハードコード可）
- `company_id`: **12669261**
- 法人口座 `walletable_id`: **4772220**（GMOあおぞらネット銀行 法人 ***8882 ＝デビット引落元）
- 勘定科目 `account_item_id`:
  - 仮払金 = **1051645906** / 旅費交通費 = **1051646033** / 立替金 = 1051645896
  - 役員報酬 = 1051646018 / 雑費 = 1051646058 / 未収入金 = 1051645904 / 前払費用 = 1051645898
- `wallet_txns.status`: 1=明細のみ(未記帳) / 2=取引登録済み。`rule_matched=true`=自動仕訳ルール適用済み。
  - **自動ルールの誤分類に注意**: 「タカギシユンスケへの振込」が一律「役員報酬」に化ける既知の罠あり。臨時の前渡しは仮払金が正しい。
- `tax_code`: 2 = 対象外（仮払金・立替金・振替など）。
- deal 修正 `PUT /api/1/deals/{id}`: **details は全置換**。既存行を直すなら `id` を指定。**`payments` も必ず body に含める**（省略すると決済情報が消える）。
- 領収書添付: `freee_file_upload`(POST /api/1/receipts, `document_type=receipt` / `qualified_invoice` / 金額・発行日・取引先メタ付き)でファイルボックスへ上げて `receipt.id` を得る → `manual_journal`/`deal` を **PUT** して `receipt_ids:[...]` を紐付ける（PUT は details 全置換なので既存 detail を `id` ごと再送）。JR等の登録番号入り領収書は `qualified_invoice="qualified"`。**領収書PDFはファイル名と中身が逆のことがある＝中身で判断**。

## 4. 会計処理パターン
- **(C) 前渡し**:
  - 振込時: (借)仮払金 / (貸)普通預金
  - 精算時: (借)旅費交通費=実費 / (貸)仮払金。余りは仮払金残（返金 or 次回相殺）。
  - ⚠️ 役員個人へ臨時に渡す金を**「役員報酬」にしない**。定期同額給与でないと損金不算入＋源泉徴収の問題が出る。仮払金が正。
- **(A) 直接決済**: wallet_txn(status1) を旅費交通費で取引登録。領収書を receipt として添付。
- **(B) 立替**: Gフォーム回答 → レシートは領収書保管袋へ → 経理が精算スプシ未チェック分を人別合計→送金→精算履歴に振込受付書添付・元シートにチェック。
- **構想日本案件**の経費は先方に請求可能（plural-reality「構想日本への経費申請」）。出張が構想日本案件かどうかを確認する。
