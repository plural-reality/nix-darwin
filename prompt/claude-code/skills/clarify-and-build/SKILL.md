# Clarify and Build

大きいタスクの失敗のほとんどは、コードではなく **要件が曖昧なまま着手すること** で起きる。書き手(ユーザー)自身も、何が欲しいか言語化できていないことが多い。この skill は、**着手前にリサーチで選択肢を可視化し、AskUserQuestion で要件を確定させてから実装に入る** ための進行手順。CLAUDE.md の「大きい指示には ①情報をまとめる → ②ユーザーと洗練 → ③実行 のステップを踏む」の実装。

## 基本制約(設計の前提)

この skill は **本体(メイン会話)で実行される orchestrator**。`context: fork` しない。理由は 2 つの非対称性:

- **`AskUserQuestion` と `Agent`(サブエージェント起動)は本体でしか使えない**。サブエージェント内からは呼べない。
- したがって「リサーチはサブエージェントに分離(コンテキスト節約)、要件確認は本体で対話」という分担になる。リサーチャーは質問できないので、代わりに **「ユーザーに確認すべき点」を列挙して返す**。本体がそれを `AskUserQuestion` に変換する。

帰結: **質問(AskUserQuestion)は plan mode に入る前に全て済ませる**。plan mode 中は AskUserQuestion が使えない。

## いつ使うか

- 新機能・システム構築・設計・デプロイなど、着手前にリサーチと要件確定が要る大きめタスク
- ユーザーが「作りたい」と言っているが、仕様・制約・優先順位がまだ曖昧なとき
- 知らない技術・初見のコードベースが絡み、ベストプラクティスや既存実装の調査が先に要るとき

使わない場面:

- 既に要件・方針が明確なタスク(そのまま実装フローへ)
- 1 ファイル内で完結する小さい修正・リネーム・バグ fix
- 純粋な調べ物だけ(`deep-research`)、要件分解だけ(`scope-guard`)

## ワークフロー

### Phase 0 — 適用判定(秒で済ます)

タスクを 1〜2 文で要約し、「大きい/曖昧」かを判定する。小さく明確なら **この skill を抜けて普通に実装する**(過剰適用の防止)。続ける場合は unknowns(不明点)を箇条書きで仮置きする。

### Phase 1 — 並列リサーチ(サブエージェントに委譲)

unknowns を埋めるため、関係する軸だけを **1 メッセージ内で並列起動** する(独立なら必ず並列)。

| 軸 | サブエージェント | 取りに行くもの |
|---|---|---|
| 外部知識 | `web-search-researcher` | ベストプラクティス・最新情報・規約・既知の落とし穴 |
| コードの在処 | `codebase-locator` | 関連ファイル・ディレクトリの location |
| コードの仕組み | `codebase-analyzer` | 既存実装の data flow・アーキテクチャ |
| 模倣可能な実装 | `codebase-pattern-finder` | 踏襲できる既存パターン(コード片) |
| 横断探索 | `Explore` | 上記が混ざる広い探索 |
| 過去文脈 | `pendant-context` | ユーザーの過去会話・決定・類似タスク |

起動プロンプトは下記「リサーチャー起動契約」に従う。コードが絡まないタスクなら外部知識・過去文脈だけでよい。

### Phase 2 — 集約と曖昧点の抽出(本体)

戻ってきた結果を統合し、次の 3 つに整理する:

- (a) 判明した事実 + 出典
- (b) 取りうる選択肢とトレードオフ
- (c) **ユーザーにしか決められない曖昧点** ← Phase 3 の質問の種

### Phase 3 — 要件の反復確定(本体 / AskUserQuestion)

`AskUserQuestion` で要件を詰める(1 回最大 4 問、各 2〜4 択、header は 12 文字以内)。

- **ユーザーにしか決められないことだけ聞く**。合理的デフォルトがあるものは聞かず、宣言して進む。
- 推奨案を先頭に置き、ラベルに「(推奨)」を付け、各選択肢に根拠・トレードオフを書く。
- 必要なら複数ラウンド回す。**収束条件: 新たな曖昧点が出ず、実装に着手できる粒度に達したら次へ**。

### Phase 4 — 計画の提示・承認(EnterPlanMode → ExitPlanMode)

確定要件をもとに実装計画を組み、`EnterPlanMode` / `ExitPlanMode` で提示して承認を取る。**質問は Phase 3 で終えていること**(plan mode 中は AskUserQuestion 不可)。

### Phase 5 — 実装(既存スキルに委譲 / 並列化)

承認後に実装。skill 本体は薄く保ち、重い仕事は既存資産へ委譲する:

- 分解が要る → `scope-guard`(Issue/ブランチ分割)
- 技術設計が要る → `cl:create_plan`
- 振る舞いを変える → `tdd`(テストファースト)
- 独立した作業は **サブエージェントで並列実装**

コードは CLAUDE.md の関数型ドグマに従う(immutable / 式中心 / Result 型 / no for-while)。

### Phase 6 — 検証(委譲)

完了を主張する前に `verify-completion` の規律で検証コマンドを実行し出力を確認する。lint/format は `fix`。

## リサーチャー起動契約

Phase 1 の各サブエージェントに渡すプロンプトはこの構造を取る(`empirical-prompt-tuning` の入出力契約に倣う):

```
# [調査テーマ] の調査

あなたは [役割: 例 Web リサーチャー / コードアナライザー] です。

## 文脈
[ユーザーの元の要求 + これまでに分かっていること]

## 調べてほしいこと
[具体的な問い 3〜5 個。曖昧にしない]

## 出力形式(必須)
- 結論: テーマごとに「結論 + 根拠(出典 URL / file:line)」
- 事実と推測の区別: 確証のあるものと推定を明示的に分ける
- ユーザーに確認すべき点: あなたが判断できない / ユーザーの意思で決まる論点を箇条書き ← 重要
- 推奨: 現時点での推奨案(あれば)
```

最後の「ユーザーに確認すべき点」が Phase 3 の AskUserQuestion の素材になる。リサーチャーは質問できないので、ここで論点だけ挙げさせる。

## 委譲先マップ

| フェーズ | 委譲先 | 備考 |
|---|---|---|
| リサーチ | web-search-researcher / codebase-* / Explore / pendant-context | 並列。新規エージェントは作らない |
| 要件確定 | (本体)AskUserQuestion | サブエージェント不可 |
| 計画承認 | EnterPlanMode / ExitPlanMode | 質問はこの前に済ます |
| 分解 | scope-guard | Issue/ブランチ |
| 設計 | cl:create_plan | 技術詳細 |
| 実装 | tdd / general-purpose サブエージェント | 並列化 |
| 検証 | verify-completion / fix | 証拠ベース |

## Red flags(合理化に注意)

| 出てくる合理化 | 実態 |
|---|---|
| 「リサーチをこの skill 自体を fork して中でやろう」 | fork すると AskUserQuestion が使えなくなる。本体で動かし、リサーチだけサブエージェントに投げる。 |
| 「plan を出してから質問しよう」 | plan mode 中は AskUserQuestion 不可。質問は Phase 3(plan 前)で終える。 |
| 「曖昧でもとりあえず実装、後で直す」 | この skill の存在理由を否定している。Phase 3 まで実装に入らない。 |
| 「全部この skill 内に書こう」 | 巨大化して既存(shibaki/create_plan)と重複。薄い orchestrator に保ち委譲する。 |
| 「合理的に決まることまで全部 AskUserQuestion で聞こう」 | 質問過多はうざい。デフォルトがあるものは宣言して進む。 |
| 「リサーチは 1 エージェントで順番に」 | 独立な軸は並列起動(1 メッセージ内で複数 Agent)。 |
| 「小さいタスクにも一応適用」 | Phase 0 で抜ける。過剰適用は害。 |

## 関連

- `deep-research` — リサーチ単体を深くやるとき(この skill の Phase 1 を単独で)
- `scope-guard` — 要件確定後の Issue/ブランチ分解
- `cl:create_plan` — 確定要件からの技術設計
- `tdd` / `verify-completion` / `fix` — 実装・検証フェーズの委譲先
- `empirical-prompt-tuning` — この skill 自体を実行者に動かして堅牢化するとき
