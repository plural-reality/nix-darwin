## 実行手順

### Step 1: 募集要項の読み込みと構造化

ユーザーから募集要項を受け取る。形式は以下のいずれか:
- PDFファイルのパス
- URL（WebFetchで取得）
- テキスト貼り付け

以下の項目を抽出し、構造化して確認を取る:

```
=== 案件情報 ===
業務名: [抽出]
発注自治体: [抽出]
テーマ/分野: [住民参加/総合計画/行革/etc.]
見積限度額: [抽出]
委託期間: [抽出]
提出締切: [抽出]
評価基準と配点: [抽出。なければ標準配点を適用]
提出書類一覧: [抽出]
特記事項: [制約条件、必須要件など]
==================
```

**確認ポイント**: 抽出結果をユーザーに提示し、誤りがないか確認してから次に進む。

### Step 2: 類似案件の自動選定

Scrapbox の plural-reality プロジェクトから、テーマ・地域・規模が近い過去案件を検索する。

#### 検索戦略（並列実行）

```bash
# テーマ・地域・実績で全文検索。cosense-fetch -s が connect.sid 認証と
# URL エンコードを隠蔽する（結果は元と同じ検索 JSON）。
cosense-fetch -s "THEME_KEYWORD"    # テーマ（住民参加 / 総合計画 / 行財政改革 等）
cosense-fetch -s "REGION_KEYWORD"   # 地域・自治体名
cosense-fetch -s "答申書"           # "自分ごと化会議" 等の実績ページ特定
```

#### 参照すべき固定ページ

以下のページは毎回必ず参照する（2hop検索で深掘り）:

```bash
# 固定ページを2hopで深掘り。cosense-fetch -h 2 は cosense-context-proxy 経由で
# Smart Context (export-2hop-links) を raw text 取得する（認証も proxy 側に集約）。
cosense-fetch -h 2 "構想日本の契約単価（実データ）（2026/2/17）"
cosense-fetch -h 2 "プロポーザル企画提案書の統一フォーマット"
cosense-fetch -h 2 "公共調達の基礎知識"
```

#### 類似案件の選定基準

以下の優先順位で3〜5件を選定:

1. **テーマの一致**: 住民参加、総合計画、行財政改革、公共施設等
2. **自治体規模の近さ**: 人口規模が近い自治体
3. **地域の近さ**: 同一都道府県 > 同一地方 > 全国
4. **直近の実績**: 新しい案件を優先

各案件から引用する情報:
- 自治体名、実施年度
- 参加者数、提案数
- 具体的な成果指標（応募率、意見数、採択率等）
- 担当者からの評価コメント（あれば）

### Step 3: 企画提案書ドラフト生成

以下のテンプレート構造に沿い、募集要項の内容と実績データで埋める。

#### 文章生成の原則

**natural-writing スキルの原則を全面適用する。** 特に:
- 「重要な役割を果たしている」「大きな意義を持つ」は禁止。代わりに数字と事実を入れる
- 「近年、〜が注目を集めています」で始めない。具体的な事実から入る
- 「多岐にわたる」「包括的な」「革新的な」の連続は空洞。何をするか具体的に書く
- 評価者は短時間で複数の提案書を読む。メリハリを付け、重要な箇所に厚く書く

**評価基準の文言に合わせたキーワード配置**: 評価基準の採点項目に使われている文言を、対応するセクションに意識的に含める。評価者がチェックリスト的に採点する場合に引っかかるようにする。

### Step 4: ユーザーレビューと修正

生成したドラフトをセクションごとに提示し、以下を確認:

1. **事実関係の正確性**: 実績数値、人名、自治体名に誤りがないか
2. **トーンの適切さ**: 自治体向けの文書として適切か
3. **評価基準との整合**: 配点構造に対して適切にボリューム配分されているか
4. **追加・削除**: 盛り込むべき情報の過不足

ユーザーの修正指示を反映し、最終版を確定する。

### Step 5: 出力とScrapbox記録

#### ファイル出力

最終版を以下のパスに出力:

```
~/Downloads/[自治体名]/プロポ10倍/[書類名]_[自治体名]_[業務名略称].docx
```

**重要: 自治体提出書類は必ずdocx形式で出力する。** pandocで変換:
```bash
pandoc -f markdown -t docx -o "企画提案書_[自治体名]_[業務名].docx" draft.md
```

#### Scrapbox記録

最終版をScrapboxの plural-reality プロジェクトに保存する。

```bash
# 1. CSRFトークンの取得
CSRF_TOKEN=$(curl -s "https://scrapbox.io/api/users/me" \
  -H "Cookie: connect.sid=$SCRAPBOX_SID" | jq -r '.csrfToken')

# 2. インポート用JSONの構築
# タイトル: "[自治体名] [業務名] プロポーザル企画提案書 (YYYY/MM/DD)"
# ページ本文をScrapbox記法に変換して投入

# 3. ページ作成（importAPI）
curl -X POST "https://scrapbox.io/api/page-data/import/plural-reality.json" \
  -H "Cookie: connect.sid=$SCRAPBOX_SID" \
  -H "X-CSRF-TOKEN: $CSRF_TOKEN" \
  -F "file=@/tmp/scrapbox-import.json;type=application/json"
```

**Scrapboxページのタグ付け:**
- `#プロポーザル`
- `#[自治体名]`
- `#[テーマ]`
- `#プロポ10倍`

## 品質チェックリスト

生成後、以下を自動チェックする:

### 内容面
- [ ] 募集要項の全要求事項に対応するセクションが存在する
- [ ] 評価基準の配点比率に応じたボリューム配分になっている（実績セクションが最も厚い）
- [ ] 具体的な数値が各セクションに含まれている（「多くの」「大幅に」は不可）
- [ ] 類似案件が3件以上挙げられている
- [ ] 見積額が限度額の85〜90%の範囲に収まっている

### 文章面（natural-writing準拠）
- [ ] 「重要な役割を果たしている」「画期的な」「革新的な」等の空疎な修飾語がない
- [ ] 「近年〜が注目を集めています」で始まるセクションがない
- [ ] 受動態の連続がない（「〜されている」「〜と考えられている」の連鎖）
- [ ] 「〜において」が一段落に3回以上出現していない
- [ ] 固有名詞と具体的な数値で事実を述べている

### 構造面
- [ ] 提出書類一覧の全項目に対応する成果物がある
- [ ] スケジュールが委託期間内に収まっている
- [ ] 体制図のメンバーが実在する（構想日本の公開情報と整合）

## このスキルがやらないこと

- 定型書類（参加申込書、誓約書等）の作成
- 登記簿謄本・定款の取得
- 電子入札システムへの提出操作
- 他社の提案書の分析・比較（倫理上）
